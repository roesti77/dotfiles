# Global Claude-Code Memory

Globale Konventionen + Pointer, die über alle Projekte hinweg gelten.

## Arbeitskonventionen

- **Immer ein Issue zu einem PR.** GitHub-Issue (Intent/Context/Acceptance) vor jedem PR, PR referenziert `Closes #N`. Projektübergreifend. → ausführbarer Ablauf: Skill `pr-flow`.
- **Source-Code + Code-Kommentare immer Englisch** (allgemeine Stil-/Sprachregeln: Skill `coding-rules`). Repo-spezifische Sprach-Konventionen für Doku/READMEs beachten — manche Repos sind durchgängig englisch, andere erlauben deutsche Doku.
- **GitHub-Issues auf „in progress" setzen, sobald sie angefasst werden.** Projects-Board-Status falls vorhanden, sonst `status: in-progress`-Label. Projektübergreifend. → Skill `pr-flow`.
- **Kein Over-Engineering — die einfachste tragfähige Lösung gewinnt (YAGNI).** Nichts bauen, was keine gestellte Anforderung verlangt: keine spekulativen Abstraktionen (Interface mit einer Implementierung), keine Konfigurierbarkeit ohne zweiten Wert, keine Durchreich-Wrapper, kein Retry/Caching/Fallback ohne belegtes Problem. Lösung deutlich größer als das Problem → anhalten und nachfragen statt weiterbauen. Projektübergreifend. → harte Regeln + Prüffrage: Skill `coding-rules`; Review: Agent `code-quality-pragmatist` / `/simplify`.
- **Bei Fehlern/Incidents: Ursachenanalyse vor Aktion — NICHT raten.** Kein trial-and-error vor belegter Root-Cause: Symptom → Hypothese → gezielter Test/Logs → Ursache → Fix; Variablen einzeln isolieren; Tool-Diagnose zuerst. Projektübergreifend. → Methode + Checkliste: Skill `rca`.
- **Approve-Verdikt = formales `gh pr review --approve`, nicht nur `--comment`** (sonst zählt es nicht aufs Merge-Gate). Ausnahme: eigene PRs → `--comment` (Self-Approve = HTTP 422). Projektübergreifend. → Details: Skill `pr-review`.
- **Review nur (re-)requesten, wenn es etwas Neues für den Reviewer gibt** — kein Status-Ping. Vorm Taggen prüfen, ob er schon angefragt ist (`gh pr view --json reviewRequests,reviews`). Projektübergreifend. → Details: Skill `pr-review`.
- **Immer in Git-Worktrees committen, NIE im Hauptcheckout.** Parallele Sessions hinterlassen fremde WIP im Hauptcheckout — dessen Branch nie annehmen/wechseln/stashen/resetten; vor dem Worktree den Branch + `git status` des Hauptcheckouts prüfen. Live-Cluster-Aktionen laufen branch-unabhängig. Projektübergreifend. → Commands + Ablauf: Skill `pr-flow`.
- **„Vorhanden/aktiv" erst behaupten nach Verifikation des enabled-Flags + (wenn erreichbar) des Live-States — Config-Sektion ≠ deployt.** Eine Config, die auf etwas zeigt (`*_url`, `[[plugin]]`, eine `*.enabled`-Sektion), beweist NICHT, dass das Ziel läuft. Projektübergreifend. → maßgebliche Schalter, Live-Checks + Mimir-Beispiel: Skill `rca`.
- **Externe App/Version NIE aus Modell-Wissen pinnen — echte aktuelle Upstream-Version prüfen und base/Prod-Parität spiegeln.** Modell-Versionswissen ist per Knowledge-Cutoff veraltet (oft eine ganze Major daneben). Vor jedem Einbinden/Bumpen einer externen Komponente (Helm-Chart, Image, Operator, CRD-Bundle, CLI) die reale Version abfragen (`helm search repo --versions`, `gh release`, `skopeo list-tags`) und gegen die Referenz-Wahrheit (base-Substrat/Prod) pinnen. Ursache: `talos-platform-apps#471` (lokales ArgoCD v2.13.0 aus Modell-Wissen statt base v3.3.2). Projektübergreifend. → Lookups + Paritäts-/Verifikationsablauf: Skill `upstream-version`.
- **Multi-Step-Arbeit mit Todo-Listen führen; vor jedem Todo-Item nach Möglichkeit ein Context-Compact.** Bei jeder Aufgabe mit mehr als einem Schritt eine Todo-Liste anlegen + pflegen (TaskCreate/TaskUpdate: Item beim Start auf `in_progress`, nach Abschluss auf `completed`; pro Item EIN Fokus) — gibt Robert den Überblick und hält den Fokus (Gegenmittel gegen das „Auseinanderlaufen" über lange + parallele Sessions). **Vor dem Start jedes Todo-Items nach Möglichkeit ein Context-Compact ausführen**, damit der Kontext schlank bleibt und der Fokus aufs aktuelle Item liegt; wo ein echtes Compact nicht selbst auslösbar ist, mindestens an jeder Todo-Grenze einen knappen Stand-Checkpoint emittieren (was fertig / was als Nächstes) — so ist die Compaction-Grenze sauber. Gilt projektübergreifend.
- **Jeder Commit muss explizit signiert werden.** Lokale Commits zwingend `git commit -S` (GPG/SSH-Signing → GitHub `Verified`); Commits über die GitHub-API/Web-Flow (`gh api .../contents`, `create_or_update_file`) werden serverseitig mit GitHubs Key signiert und gelten ebenfalls als `Verified` — bevorzugt nutzen, wenn lokales Signing nicht greift. Nie unsignierte Commits pushen. Projektübergreifend.
- **🚫 ABSOLUT: NIEMALS mit GitHub-Repos interagieren, in denen Robert nicht Member ist.** Kein PR, kein Issue, kein Kommentar, kein Review, kein Fork, kein Branch/Push — KEINE schreibende oder kommunizierende Aktion auf Repos außerhalb von Roberts Orgs. Upstream/öffentliche Repos (Drittanbieter-Charts/-Projekte) sind **reine Lese-/Pull-Quellen** (`helm pull`, `clone`, `gh ... view/list`, read) — niemals Ziel einer Aktion. Upstream-Fixes/Beiträge NUR nach ausdrücklicher, fallweiser Freigabe durch Robert, nie eigenmächtig in seinem Namen. Vor jeder `gh pr/issue/api`-Schreibaktion das `--repo`-Ziel prüfen: gehört es nicht zu einer Robert-Org → ABBRUCH + nachfragen. Keine Ausnahme, projektübergreifend.

## Second Brain (Obsidian-Vault, PARA)
